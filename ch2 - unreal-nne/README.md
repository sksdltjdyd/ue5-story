> 💡Deep Learning이란?
> 
1. 딥러닝 < 머신러닝 < 인공지능
2. 딥러닝:
    - 인공신경망을 기반으로 데이터를 학습하고 규칙을 찾아내고 판단하는 머신러닝의 한 분야
3. 머신러닝:
    - 정해진 규칙없이 데이터로부터 학습을 하고 스스로 규칙을 찾아내는 인공지능 분야
4. 인공지능:
    - 사람의 일을 기계가 대신하는것(프로그래머가 작성하는 간단한 알고리즘도 여기에 포함)
5. 딥러닝은 라벨링(분류)후 인공신경망을 이용한 방식. 목표 변수를 설정하고 지도학습을 하는 방식
    - 지도학습 기본 구조 : y = f(x) + $ --> 함수를 통해 x로부터 y를 예측, 추정하는것. x와 y 사이 함수f를 추정하는 과정이 학습
    - x : input data, features, variables, 독립변수, 설명 변수 / y : output data, label, 종속변수, 반응변수
    - (xtrain, ytrain) : 학습에 사용할 데이터셋 / (xtest, ytest) : 성능 평가에 사용할 데이터셋 --> train/test는 중복되면 안됨

> 💡Tensor란 무엇인가?
> 
1. 텐서란 넘파이의 배열과 유사한 자료형으로 다차원 행렬 데이터타입
2. 사용 방법도 넘파이와 비슷하다
    - scalar : 상수, 하나의 값
    - vector : 여러개의 값을 가진 1차원 데이터
    - matrix : 행렬, 행과 열로 이루어진 2차원 데이터
    - tensor : n차원의 행렬 형태로 표현

> 💡ONNX란?
> 
1. ONNX의 개념:
    - AI에 활용되는 FBX 파일
    - 모델러들이 각자의 툴로 제작한 모델을 fbx라는 범용 포맷으로 바꿔서 활용하듯이, AI에서는 파이토치, 텐서플로우 등 각자 편한 툴로 AI를 만든 후 onnx로 변환해서 활용한다
2. ONNX가 필요한 이유:
    1. 파이썬 종속성 탈피: 파이토치는 파이썬이라는 무거운 환경에서 돌아간다. 그렇기에 게임 클라이언트에 파이썬을 통째로 내장할 수는 없다. ONNX 포맷으로 구워내면 파이썬 없이 C++, C#, 심지어 웹에서도 AI를 돌릴 수 있다
    2. 압도적인 가벼움과 속도: 파이토치 안에는 학습을 위한 무거운 기능이 잔뜩 있다. ONNX 파일로 변환하면 이러한 무거운 기능은 전부 날아가고 오직 추론만을 위한 뼈대만 남아 용량이 줄고 실행속도가 매우 빨라진다
    3. 하드웨어 가속 최적화: 유저들의 컴퓨터의 하드웨어 스팩은 제각각이다. ONNX 모델 하나만 만들어두면, 각 기기의 하드웨어 가속기를 알아서 활용해 최적의 속도를 내준다
3. ONNX vs onnxruntime:
    - onnx패키지: 파일의 포맷 자체를 다루는 도구. 모델이 규칙에 맞게 잘 변환되었는지 검사 할때 쓴다
    - onnxruntime패키지: .onnx 파일을 실제로 실행시켜주는 엔진. 언리얼 NNE 엔진의 내부에도 이 패키지가 있다

> 💡ONNX Export
> 
1. 더미 입력 (dummy_input): "가짜 데이터로 길 뚫어주기"
    - 파이토치는 코드가 실행될 때 실시간으로 데이터의 흐름을 파악하는 유연한 값이다. 반면 onnx와 언리얼 엔진은 형태가 고정된 그래프를 요구한다
    - 역할 : 모델에게 가짜 쓰레기 값을 한번 통과시킨다. 그러면 파이토치가 이 데이터가 모델 안에서 어떻게 쪼개지고 합쳐지는지 그 경로를 추적해서 onnx파일로 구워 낸다
2. 입출력 이름 (input_names, output_names): "C++ 바인딩용 소켓 이름"
    - 추후 언리얼에서 사용할때 문제가 없도록 변수 명을 명확히 명시해야 함
    - 역할: 이름을 안 지어주면 파이토치가 자기 마음대로 1, 2, input.1 같은 랜덤한 숫자로 입출력 노드 이름을 정해버림. 나중에 언리얼 C++에서 데이터를 밀어 넣을 때 SetInput("input_image", ...)처럼 명확한 변수명으로 접근하기 위해 반드시 지정
3. 동적 차원 (dynamic_axes): "가변 배열 세팅하기"
    - 역할: 원래 ONNX는 dummy_input으로 넣은 크기(예: 이미지 1장) 그대로 굳어버림. 하지만 게임에서는 프레임마다 AI가 1마리일 수도, 5마리일 수도 있다. "0번째 차원(배치 사이즈)은 1장이 들어올지 5장이 들어올지 모르니까 고정하지 말고 열어둬!"라고 설정하는 기능.
4. 오프셋 버전 (opset_version): "언리얼 호환성 맞추기"
    - 역할: ONNX도 문법이 계속 업데이트됩니다. 너무 최신 버전(opset 17 등)으로 구워버리면, 언리얼 엔진 내부에 있는 NNE(ONNX Runtime)가 구버전일 경우 "나 이거 못 읽어!" 하고 튕겨버립니다. 보통 안정성을 위해 opset_version=14 정도를 많이 씁니다

> 💡Shape가 왜 언리얼에서 중요한가
> 
1. 파이썬의 NumPy나 PyTorch는 [1, 3, 224, 224] 같은 다차원 배열을 유연하게 다루지만, 언리얼 C++의 TArray는 다차원 구조를 이해하지 못함
2. 언리얼 C++로 데이터를 밀어넣을 때는 데이터를 1차원으로 쫙 펴서(Flatten) 밀어넣어야 한다. [3, 224, 224] 형태의 이미지는 3 x 224 x 224 = 150528개의 원소를 가진 1차원 배열로 들어간다
3. 모델의 정확한 Shape을 모르면 C++ 메모리를 할당할 때 엔진이 터져버린다
